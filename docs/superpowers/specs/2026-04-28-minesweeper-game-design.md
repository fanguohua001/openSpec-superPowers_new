# 扫雷休闲游戏设计

## 目标

在现有 RuoYi-Vue 管理系统中新增一个轻量级“扫雷”休闲游戏页面，作为前端可直接游玩的单页功能，并通过系统菜单接入。首版聚焦经典扫雷玩法、清晰反馈和稳定的前端实现，不引入后端业务状态。

## 已确认方案

采用“轻量前端游戏 + 菜单接入”方案：

- 在 `ruoyi-ui/src/views/game/minesweeper/` 新增扫雷页面。
- 将扫雷核心逻辑拆为纯 JavaScript 模块，便于测试。
- 在 `sql/ry_20250522.sql` 新增“休闲游戏 / 扫雷”菜单初始化数据。
- 页面沿用 Vue 2、Element UI 和 RuoYi 的布局习惯，保持管理系统内的统一体验。

## 范围

本次包含：

- 新增扫雷页面，支持在系统菜单中进入。
- 支持初级、中级、高级三种难度。
- 支持自定义行数、列数和雷数的基础限制。
- 支持左键翻开格子。
- 支持右键或长按切换旗标。
- 支持首次点击安全，首次翻开不会踩雷。
- 支持空白区域自动展开。
- 支持显示剩余雷数、计时、已翻开数量和当前局状态。
- 支持胜利、失败、重新开始。
- 失败后显示全部地雷和错误旗标。
- 胜利后自动标记剩余地雷。
- 使用纯函数测试覆盖棋盘生成、首次安全、翻开扩散、插旗、胜负判断等核心规则。
- 运行前端测试或等价 Node 测试，并运行前端构建验证。

本次不包含：

- 后端排行榜、历史战绩、个人最佳成绩持久化。
- 多人游戏、联机对战。
- 皮肤市场、音效系统。
- 修改登录、首页或现有业务模块。
- 引入额外游戏引擎。

## 用户体验

页面放在 RuoYi 内容区内，结构保持紧凑：

- 顶部为工具栏，包含难度选择、行列雷数输入、重新开始按钮和状态摘要。
- 中部为扫雷棋盘，棋盘根据列数使用 CSS Grid 渲染。
- 底部或侧边提供局内统计，包括计时、剩余雷数、已翻开格子、当前状态。
- 格子尺寸在桌面端保持稳定，在窄屏下允许横向滚动，避免压缩导致误触。
- 格子视觉区分未翻开、已翻开、旗标、地雷、爆炸点和错误旗标。

交互规则：

- 点击未翻开的普通格子会翻开该格。
- 翻开数字为 `0` 的格子时，自动展开相邻安全区域。
- 右键未翻开的格子切换旗标，并阻止浏览器默认菜单。
- 已插旗格子不能被左键翻开。
- 游戏结束后棋盘不可继续操作，只能重新开始。
- 切换难度或自定义参数后会立即开启新局。

## 游戏规则

难度预设：

- 初级：`9 x 9`，`10` 个地雷。
- 中级：`16 x 16`，`40` 个地雷。
- 高级：`16 x 30`，`99` 个地雷。

自定义限制：

- 行数范围：`5` 到 `24`。
- 列数范围：`5` 到 `40`。
- 雷数范围：`1` 到 `行数 * 列数 - 9`。

首次点击安全：

- 首次翻开时再布雷。
- 首次点击格及其相邻八格不放置地雷。
- 如果棋盘过小导致安全区过大，则仍保证首次点击格不是地雷，并在可用格内布满所需地雷。

胜负判断：

- 翻开地雷立即失败。
- 所有非地雷格子都被翻开时胜利。
- 插旗数量本身不作为胜利条件。

## 技术设计

### 前端页面

新增 `ruoyi-ui/src/views/game/minesweeper/index.vue`：

- 使用 Vue 2 Options API。
- 使用 Element UI 的按钮、单选组、输入数字和标签组件。
- 页面不调用后端接口。
- 页面状态由纯游戏核心返回的新状态驱动，避免把规则散落在模板方法中。

### 游戏核心

新增 `ruoyi-ui/src/views/game/minesweeper/minesweeper.js`：

- `createGame(options)`：创建新局。
- `revealCell(game, row, col)`：翻开格子。
- `toggleFlag(game, row, col)`：切换旗标。
- `getNeighbors(row, col, rows, cols)`：获取相邻坐标。
- `buildBoard(game, firstRow, firstCol)`：按首次点击安全规则布雷并计算数字。

核心模块保持纯 JavaScript，不依赖 Vue、Element UI 或浏览器 DOM，便于通过 Node 直接测试。

### 测试

新增 `ruoyi-ui/src/views/game/minesweeper/minesweeper.test.js` 或同目录测试脚本：

- 创建棋盘时尺寸和初始状态正确。
- 首次翻开不会踩雷。
- 首次安全区不布雷，空间不足时至少保证点击格安全。
- 数字格显示相邻地雷数量。
- 空白格翻开会扩散相邻安全区域。
- 插旗格不会被翻开。
- 踩雷后状态为失败，并记录爆炸格。
- 翻开全部安全格后状态为胜利，并标记剩余地雷。

如仓库没有现成前端测试框架，则新增一个轻量 Node 测试脚本，只测试纯逻辑模块，不引入 Jest/Vitest 等新依赖。

### 菜单接入

在 `sql/ry_20250522.sql` 增加初始化菜单：

- 父级目录：`休闲游戏`。
- 子菜单：`扫雷`。
- 组件路径：`game/minesweeper/index`。
- 权限标识：`game:minesweeper:view`。
- 图标优先使用现有 SVG 图标，例如 `bug` 或 `question`，不新增图标资源。

菜单 ID 需要避开现有 `sys_menu` 初始化数据中的 ID，采用未占用区间。

## 验证标准

- OpenSpec 变更工件完整且 `openspec validate <change-name> --no-color` 通过。
- 扫雷核心逻辑测试通过。
- `npm run build:prod` 在 `ruoyi-ui` 下通过。
- 菜单 SQL 中存在“休闲游戏 / 扫雷”初始化数据，组件路径能映射到新增页面。
- 页面代码不引入额外运行时依赖，不影响现有审批、系统管理和工具页面。

## 风险与取舍

- 不做后端排行榜，避免把休闲游戏首版扩展成跨模块业务功能。
- 不新增图标资源，降低构建和资源注册风险。
- 自定义棋盘允许横向滚动，优先保证格子可点击性和布局稳定，而不是强行压缩到所有视口。
- 测试重点放在纯逻辑模块；页面渲染通过构建和必要的浏览器手工检查验证。
