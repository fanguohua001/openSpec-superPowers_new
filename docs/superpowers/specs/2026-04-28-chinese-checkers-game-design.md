# 中国跳棋休闲游戏设计

## 目标

在现有 RuoYi-Vue 管理系统中新增一个轻量级“中国跳棋”休闲游戏页面，作为前端可直接游玩的本地双人游戏，并通过系统菜单接入。首版聚焦经典双人对向玩法、合法移动提示、连续跳跃和胜负判断，不引入后端业务状态、排行榜或电脑对手。

## 已确认方案

采用“本地 2 人中国跳棋 + 菜单接入”方案：

- 在 `ruoyi-ui/src/views/game/chinese-checkers/` 新增中国跳棋页面。
- 将棋盘坐标、棋子状态、合法移动、连续跳跃和胜负判断拆为纯 JavaScript 模块，便于测试。
- 新增菜单增量 SQL，让已有数据库可以补充“休闲游戏 / 跳棋”入口。
- 页面沿用 Vue 2、Element UI 和 RuoYi 的内容区布局，保持与现有扫雷页面相近的管理台体验。

## 范围

本次包含：

- 新增中国跳棋页面，支持在系统菜单中进入。
- 支持本地双人对局，双方各 `10` 枚棋子。
- 使用中国跳棋星形棋盘的 121 个有效落点。
- 支持点击选择当前回合棋子。
- 支持高亮当前棋子的所有合法落点。
- 支持相邻一步移动。
- 支持按直线隔一子跳到空位。
- 支持一次操作中连续跳跃，玩家可在每次跳跃后继续选择下一跳或结束回合。
- 支持回合切换、当前玩家提示、步数统计和重新开始。
- 支持胜负判断：一方全部棋子进入对面目标营地时获胜。
- 使用纯函数测试覆盖棋盘生成、初始摆放、一步移动、跳跃、连续跳跃、非法移动、回合切换和胜负判断。
- 运行前端规则测试和前端构建验证。

本次不包含：

- 电脑对手或自动提示最佳走法。
- 2-6 人局、组队模式或自定义人数。
- 后端排行榜、历史战绩、个人最佳成绩持久化。
- 联机对战、观战、悔棋记录持久化。
- 音效、皮肤市场或新增图标资源。
- 修改登录、首页、审批流、扫雷或现有业务模块。

## 用户体验

页面放在 RuoYi 内容区内，结构保持紧凑：

- 顶部显示标题、当前回合、步数、对局状态和重新开始按钮。
- 主区域显示星形棋盘，棋盘采用坐标点布局渲染，落点尺寸稳定并适配桌面和移动端。
- 侧边显示双方状态，包括玩家名称、棋子颜色、剩余未到达目标营地的棋子数量和当前胜负状态。
- 当前回合玩家的棋子可点击选中，非当前回合棋子不可移动。
- 选中棋子后高亮合法落点，一步移动和跳跃落点用不同视觉状态区分。
- 完成一步移动后自动切换回合。
- 完成跳跃后，如果当前棋子仍有后续跳跃落点，则保持选中并提示可以继续跳跃或结束回合。
- 对局结束后显示胜利提示，并禁止继续移动棋子，只允许重新开始。

## 游戏规则

### 棋盘与阵营

- 棋盘采用中国跳棋标准六角星形有效落点，共 `121` 个坐标点。
- 首版启用两组相对阵营，每方 `10` 枚棋子。
- 玩家一从上方营地出发，目标为下方相对营地。
- 玩家二从下方营地出发，目标为上方相对营地。
- 非启用阵营的落点仍作为可经过和可停留的棋盘点。

### 移动规则

- 棋子可以移动到六个相邻方向上的空落点。
- 棋子可以沿六个方向跳过一个相邻棋子，并落到该方向下一格空落点。
- 被跳过的棋子不被移除。
- 一次回合内，如果玩家选择跳跃，则可以继续用同一枚棋子进行后续跳跃。
- 连续跳跃过程中不能改用其他棋子。
- 连续跳跃过程中，一步相邻移动不可作为后续动作。
- 玩家可在至少完成一次跳跃后主动结束回合。
- 已有棋子的落点不能作为移动目标。
- 棋子不能移动到棋盘外无效坐标。

### 胜负判断

- 任一玩家的全部 `10` 枚棋子均处于对面目标营地时，该玩家获胜。
- 获胜后对局状态变为结束，棋盘不再响应移动。

## 技术设计

### 前端页面

新增 `ruoyi-ui/src/views/game/chinese-checkers/index.vue`：

- 使用 Vue 2 Options API。
- 使用 Element UI 的按钮、标签和提示组件。
- 页面不调用后端接口。
- 页面状态由纯游戏核心返回的新状态驱动，避免把规则散落在模板方法中。
- 棋盘使用固定坐标映射渲染，保证星形布局稳定。
- 使用按钮或可聚焦元素承载棋盘落点，保留基础键盘可达性和焦点状态。

### 游戏核心

新增 `ruoyi-ui/src/views/game/chinese-checkers/chineseCheckers.js`：

- `createGame()`：创建本地双人新局。
- `selectPiece(game, position)`：选择当前玩家棋子并计算合法落点。
- `movePiece(game, from, to)`：执行一步移动或跳跃移动。
- `endTurn(game)`：在连续跳跃后主动结束回合。
- `getLegalMoves(game, position, mode)`：获取指定棋子的合法移动。
- `isWinner(game, player)`：判断玩家是否获胜。
- `createBoard()`：生成 121 个有效棋盘落点和相邻坐标关系。

核心模块保持纯 JavaScript，不依赖 Vue、Element UI 或浏览器 DOM，便于通过 Node 直接测试。

### 测试

新增 `ruoyi-ui/src/views/game/chinese-checkers/chineseCheckers.test.js` 或同目录测试脚本：

- 创建新局时棋盘有效落点数量为 `121`。
- 双方初始棋子数量均为 `10`。
- 初始棋子位于各自出发营地。
- 当前回合只能选择己方棋子。
- 相邻空位可一步移动。
- 有相邻棋子且落点为空时可跳跃。
- 跳跃后若仍有跳跃落点，可继续跳跃且不能改动其他棋子。
- 连续跳跃中不能执行一步相邻移动。
- 非法目标、被占用目标和棋盘外目标不能移动。
- 一步移动后自动切换回合。
- 连续跳跃主动结束后切换回合。
- 一方全部棋子进入目标营地后判定胜利。

如仓库没有现成前端测试框架，则沿用扫雷做法，新增轻量 Node 测试脚本，只测试纯逻辑模块，不引入 Jest/Vitest 等新依赖。

### 菜单接入

新增 `sql/chinese_checkers_menu.sql` 作为已初始化数据库的增量菜单脚本：

- 父级目录复用或补充：`休闲游戏`。
- 子菜单：`跳棋`。
- 组件路径：`game/chinese-checkers/index`。
- 权限标识：`game:chinese-checkers:view`。
- 图标优先使用现有 SVG 图标，不新增图标资源。
- 为 `common` 角色补充目录和菜单授权，避免非管理员用户登录后看不到入口。

菜单 ID 需要避开现有 `sys_menu` 初始化数据和扫雷菜单 ID。若当前 `休闲游戏` 目录已经使用 ID `123`，本次脚本应复用该目录，并为“跳棋”使用新的未占用 ID。

## 验证标准

- OpenSpec 变更工件完整且 `openspec validate <change-name> --no-color` 通过。
- 中国跳棋核心逻辑测试通过。
- `npm run build:prod` 在 `ruoyi-ui` 下通过。
- SQL 增量脚本存在“休闲游戏 / 跳棋”菜单，组件路径能映射到新增页面。
- 页面代码不引入额外运行时依赖，不影响现有扫雷、审批、系统管理和工具页面。

## 风险与取舍

- 首版不做 2-6 人局，避免棋盘阵营、回合和胜负规则一次性扩张过大。
- 首版不做电脑对手，避免把休闲游戏变成策略 AI 质量问题。
- 连续跳跃采用“继续跳跃或结束回合”的明确状态，避免误触后自动结束导致体验挫败。
- 棋盘视觉重点放在坐标稳定、合法落点清晰和移动反馈，不新增复杂动画。
- 测试重点放在纯逻辑模块；页面渲染通过构建和必要的浏览器手工检查验证。
