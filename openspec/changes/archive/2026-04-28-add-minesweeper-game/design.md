## Context

本仓库是 RuoYi-Vue 管理系统，前端位于 `ruoyi-ui`，使用 Vue 2、Vue Router、Vuex 和 Element UI。系统菜单由后端 `sys_menu` 数据驱动，前端通过 `getRouters` 获取组件路径并动态加载 `ruoyi-ui/src/views/` 下的页面。

扫雷游戏属于纯前端休闲功能，不需要后端业务数据。为了保持改动外科手术式收敛，本次只新增前端游戏页面、纯 JavaScript 规则模块、规则测试和菜单初始化 SQL，不修改登录、首页、审批流或系统管理既有行为。

## Goals / Non-Goals

**Goals:**

- 在系统菜单中提供“休闲游戏 / 扫雷”入口。
- 提供经典扫雷玩法，包括难度选择、自定义棋盘、首次点击安全、翻开、插旗、空白扩散、胜利和失败。
- 将游戏规则集中在纯 JavaScript 模块中，降低页面组件复杂度并便于测试。
- 不新增运行时依赖，不引入游戏引擎。
- 通过规则测试、前端构建和 OpenSpec 验证证明变更有效。

**Non-Goals:**

- 不实现排行榜、历史战绩或个人最佳成绩。
- 不新增后端接口、数据库业务表或持久化服务。
- 不实现多人模式、联机对战、音效系统或皮肤系统。
- 不重构现有菜单、路由、权限或其他业务页面。

## Decisions

### 使用纯前端实现

扫雷局内状态只存在于浏览器内存中，不通过后端保存。

- 选择原因：扫雷首版没有跨用户协作或持久化需求，前端实现足以覆盖需求，风险和改动范围最小。
- 备选方案：新增后端战绩表和接口。该方案能支持排行榜，但会显著扩大数据库、权限、Controller、Service 和前端 API 的变更面，不符合首版轻量目标。

### 拆分纯 JavaScript 游戏核心

新增 `minesweeper.js` 承载棋盘生成、布雷、翻开、插旗和胜负判断。Vue 页面只负责渲染、用户输入和调用核心函数。

- 选择原因：扫雷规则有较多边界条件，纯函数模块更容易通过 Node 测试覆盖，避免把规则散落在 Vue 方法和模板事件中。
- 备选方案：所有逻辑写在 `index.vue`。实现更快，但后续验证和维护成本更高。

### 首次翻开时布雷

创建新局时先生成空棋盘，首次翻开时根据点击位置布雷并计算数字。

- 选择原因：经典扫雷通常保证首次点击安全；延迟布雷是最简单可靠的实现方式。
- 备选方案：新局立即布雷，首次点击踩雷后重排。该方式需要处理已计算数字的回滚，规则更绕。

### 使用现有 Element UI 和 RuoYi 风格

页面使用 `app-container`、Element UI 表单控件、按钮和标签。棋盘使用 CSS Grid 自定义渲染，不新增 UI 组件库。

- 选择原因：保持管理系统视觉一致性，同时棋盘格子需要稳定尺寸和高频交互，自定义 CSS Grid 比表格组件更合适。
- 备选方案：使用 `el-table` 或外部棋盘库。`el-table` 对网格游戏交互不自然，外部库会增加依赖和样式适配成本。

### 菜单通过 SQL 初始化接入

在 `sql/ry_20250522.sql` 追加“休闲游戏 / 扫雷”菜单，组件路径为 `game/minesweeper/index`，权限标识为 `game:minesweeper:view`。

- 选择原因：现有审批流变更已采用 SQL 初始化菜单，沿用项目接入方式最稳定。
- 备选方案：只新增前端静态路由。该方案无法自然出现在动态菜单中，且绕过 RuoYi 权限体系。

## Risks / Trade-offs

- [Risk] 自定义大棋盘在窄屏设备上难以完整显示 → 使用固定格子尺寸和横向滚动，优先保证点击准确性。
- [Risk] SQL 菜单 ID 与未来手工菜单冲突 → 选择当前初始化脚本未占用的 ID 区间，并保持菜单注释清晰。
- [Risk] 没有现成前端单元测试框架 → 使用 Node 运行纯逻辑测试，避免为一个轻量模块引入 Jest 或 Vitest。
- [Risk] 游戏状态不可跨刷新保存 → 首版明确不做持久化，刷新后重新开始是可接受行为。

## Migration Plan

1. 部署前端构建产物。
2. 对新环境执行更新后的 `sql/ry_20250522.sql` 初始化脚本，或在已有环境手工补充相同菜单数据。
3. 给需要访问扫雷的角色分配 `game:minesweeper:view` 菜单权限。

回滚时删除新增前端文件，并删除对应 `sys_menu` 菜单数据即可。本变更不写入业务数据，不涉及数据迁移回滚。
