## Why

当前系统具备用户、角色、菜单、权限等基础管理能力，但缺少可复用的审批流能力。后续业务如果各自手写审批逻辑，会造成状态、权限、审批记录和流转规则重复实现，维护成本会快速上升。

本变更先提供轻量通用审批流，不引入 BPMN 流程引擎，以较小复杂度支持后续业务模块接入审批。

## What Changes

- 新增通用“审批管理”能力，支持维护审批流程定义、节点顺序、审批人规则和启停状态。
- 审批流程可被后续业务通过业务类型、业务主键和标题接入，审批模块不直接耦合具体业务表。
- 审批节点支持两类审批人规则：指定用户、指定角色。
- 一个审批节点可以配置多个候选审批人；首版采用或签规则，即任意一名候选审批人同意即可通过该节点。
- 支持发起审批、查询我的申请、查询待我审批、查看审批详情和审批记录。
- 支持审批动作：同意、驳回。驳回时退回上一节点继续流转；首节点驳回时退回发起人并进入驳回状态。
- 新增审批实例、审批任务和审批记录持久化，完整记录流转过程。
- 新增若依菜单、接口权限和前端页面，沿用现有 RuoYi Controller、Service、Mapper、Vue 2、Element UI 风格。
- 不引入 Flowable、Activiti、Camunda 等外部流程引擎；不支持 BPMN 画布、条件网关、会签、转办、加签、委托和超时催办。

## Capabilities

### New Capabilities

- `custom-approval-flow`: 定义通用审批流配置、审批实例流转、审批任务处理、驳回回退和审批记录查询行为。

### Modified Capabilities

无。

## Impact

- 后端模块：
  - `ruoyi-admin`：新增审批管理 Controller，对外提供流程定义、实例、任务和记录接口。
  - `ruoyi-system`：新增审批 domain、mapper、service 和流转服务。
  - `ruoyi-common`：默认不修改；仅在实现时确有必要才补充通用常量或枚举。
- 前端模块：
  - `ruoyi-ui/src/api/approval/`：新增审批相关 API 封装。
  - `ruoyi-ui/src/views/approval/`：新增流程定义、我的申请、待我审批、审批详情页面。
- 数据库：
  - 新增审批流程定义、节点定义、节点审批人、审批实例、审批任务、审批记录表。
  - 新增 `sys_menu` 菜单和按钮权限初始化数据。
- 安全与权限：
  - 沿用若依 `@PreAuthorize` 和菜单权限。
  - 待办审批查询必须只返回当前用户可处理的任务。
  - 审批动作必须校验当前用户是否为当前节点候选审批人。
- 集成边界：
  - 首版通过通用发起接口接收 `businessType`、`businessId`、`businessTitle`，后续具体业务模块负责在自身业务状态中响应审批结果。
