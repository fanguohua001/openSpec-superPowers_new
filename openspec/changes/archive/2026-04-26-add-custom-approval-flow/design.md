## Context

当前项目是 RuoYi-Vue 风格的前后端分离管理系统。后端采用 Maven 多模块，业务 Controller 主要位于 `ruoyi-admin`，系统业务位于 `ruoyi-system`；前端位于 `ruoyi-ui`，使用 Vue 2、Element UI 和 RuoYi 动态路由/菜单权限体系。

系统已有用户、角色、部门、菜单、权限、字典、操作日志和代码生成能力，但没有通用审批流。审批流需要同时触及数据库模型、权限控制、业务接入边界、前端页面和流转状态，因此先通过 OpenSpec 明确轻量方案。

首版约束来自用户确认：

- 审批流作为通用能力，后续业务模块可接入。
- 审批人规则支持指定用户、指定角色。
- 驳回退回上一节点继续流转。
- 同一节点多个候选审批人采用或签，一人同意即可通过。

## Goals / Non-Goals

**Goals:**

- 提供可维护的流程定义、节点定义和审批人配置。
- 提供业务无关的审批实例模型，通过 `businessType`、`businessId`、`businessTitle` 关联外部业务。
- 支持按流程节点顺序流转、或签通过、驳回回退上一节点。
- 支持我的申请、待我审批、审批详情和审批记录查询。
- 沿用 RuoYi 的 Controller、Service、Mapper、XML、Vue 页面和菜单权限风格。
- 让后续业务模块能够用少量代码发起审批、查询审批状态，并根据审批完成结果更新自身业务状态。

**Non-Goals:**

- 不引入 Flowable、Activiti、Camunda 等流程引擎。
- 不支持 BPMN 画布、条件网关、并行分支、会签、加签、转办、委托、抄送、催办、超时升级。
- 不在首版实现动态表单设计器。
- 不直接修改已有业务模块的业务状态机；首版只提供通用审批能力和集成边界。
- 不提供跨租户、多组织隔离模型；沿用当前若依用户和角色体系。

## Decisions

### 决策 1：采用轻量自研审批流而不是流程引擎

首版只需要线性节点、指定用户/角色审批、或签和驳回回退。用轻量表结构和服务即可覆盖，避免引入流程引擎后的依赖、数据库表、建模概念和运维复杂度。

备选方案是接入 Flowable/Activiti。它适合复杂 BPMN、并行网关和长期流程治理，但当前需求会被流程引擎概念放大，首版维护成本偏高。

### 决策 2：以 `ruoyi-system` 承载审批业务

审批 domain、mapper、service 放在 `ruoyi-system`，Controller 放在 `ruoyi-admin`。这延续现有系统管理能力的组织方式，也方便复用用户、角色、权限、字典和操作日志能力。

备选方案是新增 `ruoyi-approval` 模块。模块边界更独立，但首版功能规模尚不足以抵消 Maven 依赖、扫描配置和模块管理成本。

### 决策 3：审批模块只保存业务引用，不耦合业务表

审批实例保存 `businessType`、`businessId`、`businessTitle`、`starterUserId`、`processId` 和审批状态。审批模块不读取或更新具体业务表，业务模块通过审批结果自行处理业务状态。

备选方案是审批服务直接回调或更新业务表。它可以减少业务侧代码，但会让审批模块知道太多业务细节，后续接入新业务时容易积累条件分支。

### 决策 4：流程定义拆成流程、节点、节点审批人三层

建议数据模型：

- `approval_process`：流程定义，包含流程名称、业务类型、状态、备注、审计字段。
- `approval_node`：节点定义，包含流程 ID、节点名称、节点顺序、审批类型。
- `approval_node_approver`：节点审批人，包含节点 ID、审批人类型、用户 ID 或角色 ID。

这样可以让同一节点配置多个用户或角色，并保持查询待办审批人时的规则清晰。

备选方案是在节点表中用 JSON 保存审批人。它表少，但不利于 MyBatis 查询、权限过滤、后续统计和菜单页面维护。

### 决策 5：审批实例拆成实例、任务、记录

建议数据模型：

- `approval_instance`：一次业务审批实例，保存当前节点、总体状态和业务引用。
- `approval_task`：当前或历史待办任务，保存节点、候选审批人范围、任务状态。
- `approval_record`：不可变审批操作记录，保存操作人、动作、意见、前后节点和时间。

任务用于快速查询待我审批；记录用于详情审计。实例是业务接入时查询整体状态的主表。

### 决策 6：节点或签通过后关闭同节点其他待办

当某个节点有多个候选审批人时，系统为候选人生成待办任务或保存可展开的候选范围。任意候选审批人同意后，当前节点通过，其他同节点待办应标记为已失效或已关闭，避免重复处理。

首版推荐生成任务时尽量落到具体用户：指定用户直接生成用户任务；指定角色按当时拥有该角色的用户展开。这样待办查询简单，审批资格也可按任务校验。

备选方案是任务只保存角色范围，查询时动态判断当前角色。它对角色变更更实时，但查询和资格校验更复杂，历史审批责任也不够稳定。

### 决策 7：驳回回退上一节点，首节点驳回则退回发起人

非首节点驳回时，实例当前节点回退到上一节点，并为上一节点重新生成候选审批任务。首节点没有上一审批节点，驳回后实例进入 `rejected`，发起人可查看记录并决定是否由业务侧重新发起。

备选方案是任何节点驳回都直接终止。它更简单，但用户已明确希望退回上一节点继续流转。

### 决策 8：使用状态枚举约束流转

建议实例状态：

- `draft`：草稿或未提交，首版可不暴露。
- `pending`：审批中。
- `approved`：已通过。
- `rejected`：已驳回到发起人。
- `cancelled`：已撤销，首版可预留但不实现撤销接口。

建议任务状态：

- `pending`：待处理。
- `approved`：已同意。
- `rejected`：已驳回。
- `closed`：因或签通过、回退或实例结束而关闭。

## Risks / Trade-offs

- 角色审批人在发起后发生角色成员变化 → 首版按发起时展开为具体用户任务，保证历史稳定；后续如需实时角色，可单独扩展。
- 流程定义被修改后影响已发起实例 → 已发起实例应按发起时节点快照或当前节点 ID 继续；首版可限制已启用且已有实例的流程不允许删除节点，只允许停用流程。
- 业务模块不知道审批完成结果 → 首版提供实例状态查询和审批完成事件边界；具体业务模块在接入时负责更新自身状态。
- 或签节点候选人较多导致任务膨胀 → 指定角色人数通常可控；若后续角色人数很大，再优化为动态角色任务。
- 驳回回退上一节点可能反复流转 → 记录每次动作和节点变化，保证审计可追踪。
- 通用能力过度设计风险 → 首版只做线性流程、指定用户/角色、或签、同意/驳回，保留扩展点但不实现复杂流程能力。

## Migration Plan

1. 新增审批相关表结构和必要索引。
2. 新增审批管理菜单与权限初始化数据。
3. 部署后端审批 domain、mapper、service 和 controller。
4. 部署前端审批管理页面和 API。
5. 管理员配置流程定义并启用。
6. 后续业务模块通过通用发起接口接入审批。

回滚策略：

- 前端可通过菜单权限隐藏审批管理入口。
- 后端新接口不影响现有业务接口。
- 数据库为新增表和新增菜单数据；完整回滚可备份后删除审批相关表、菜单和权限数据。

## Open Questions

- 首个接入审批的业务模块尚未确定，因此首版先提供通用发起接口和状态查询，不绑定具体业务状态。
- 流程定义启用后的编辑策略需要实现阶段确认：建议启用后禁止删除已被实例引用的节点，避免历史实例断链。
- 是否需要“重新提交”动作由业务模块实现，审批模块首版仅记录驳回结果和历史。
