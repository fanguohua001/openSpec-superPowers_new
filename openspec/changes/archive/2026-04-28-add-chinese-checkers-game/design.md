## Context

本仓库是 RuoYi-Vue 管理系统，前端位于 `ruoyi-ui`，使用 Vue 2、Vue Router、Vuex 和 Element UI。系统菜单由后端 `sys_menu` 数据驱动，前端通过 `getRouters` 获取组件路径并动态加载 `ruoyi-ui/src/views/` 下的页面。

仓库中已存在纯前端扫雷休闲游戏，路径为 `ruoyi-ui/src/views/game/minesweeper/`，它采用页面组件、纯 JavaScript 规则模块、Node 原生断言测试和菜单增量 SQL 的组织方式。中国跳棋同样属于无后端业务状态的休闲功能，可沿用该模式，但棋盘坐标、连续跳跃和回合状态比扫雷更复杂，需要在实现前明确规则边界。

## Goals / Non-Goals

**Goals:**

- 在系统菜单中提供“休闲游戏 / 跳棋”入口。
- 提供本地双人中国跳棋玩法，双方各 `10` 枚棋子。
- 使用标准星形棋盘的 `121` 个有效落点。
- 支持棋子选择、合法落点高亮、一步移动、跳跃、连续跳跃、主动结束回合、胜负判断和重新开始。
- 将游戏规则集中在纯 JavaScript 模块中，便于通过 Node 测试覆盖。
- 不新增运行时依赖，不引入游戏引擎、AI 库或后端服务。

**Non-Goals:**

- 不实现电脑对手、自动提示最佳走法或策略评分。
- 不实现 2-6 人局、组队模式或自定义人数。
- 不实现联网对战、观战、排行榜、历史战绩或持久化。
- 不修改登录、首页、审批流、扫雷或现有业务模块。

## Decisions

### 使用纯前端本地状态

中国跳棋局内状态只存在于浏览器内存中，不通过后端保存。

- 选择原因：首版目标是本地双人休闲对局，不需要跨用户协作或持久化。纯前端实现的改动面最小，且与扫雷页面模式一致。
- 备选方案：新增后端对局表和接口。该方案能支持恢复对局或排行榜，但会扩大数据库、Controller、Service、权限和前端 API 的变更范围，不符合首版轻量目标。

### 拆分纯 JavaScript 游戏核心

新增 `chineseCheckers.js` 承载棋盘生成、初始摆放、合法移动、连续跳跃、回合切换和胜负判断。Vue 页面只负责渲染、用户输入和调用核心函数。

- 选择原因：中国跳棋规则边界多，纯函数模块更容易通过 Node 原生断言测试验证，避免把复杂规则散落在 Vue 模板事件中。
- 备选方案：所有逻辑写在 `index.vue`。实现短期更快，但后续验证、调试和扩展到多人局的成本更高。

### 采用轴向坐标生成星形棋盘

游戏核心使用六角网格轴向坐标表示棋盘点，并通过坐标条件生成六角星的 `121` 个有效落点、六个移动方向和十子营地。

- 选择原因：轴向坐标天然适配六个方向的一步移动和跳跃计算，能避免硬编码大量相邻关系。
- 备选方案：手工维护 121 个点及邻接表。该方案直观但易错，后续调整营地或布局时维护成本高。

### 连续跳跃进入专门状态

当玩家执行跳跃后，如果同一棋子仍有跳跃落点，游戏进入连续跳跃状态。此时只能继续移动同一棋子到跳跃落点，或主动结束回合。

- 选择原因：该规则清楚区分“一步移动”和“跳跃链”，能避免跳跃后误选其他棋子或误执行一步移动。
- 备选方案：每次跳跃后自动结束回合。实现更简单，但不符合中国跳棋常见玩法，也会降低游戏策略性。

### 菜单通过增量 SQL 接入

新增 `sql/chinese_checkers_menu.sql`，复用或补充“休闲游戏”目录，并新增“跳棋”菜单，组件路径为 `game/chinese-checkers/index`，权限标识为 `game:chinese-checkers:view`。

- 选择原因：现有扫雷已提供增量菜单脚本并发现过角色授权遗漏问题。本次直接补充 `sys_role_menu`，能降低非管理员用户看不到菜单的风险。
- 备选方案：只新增前端静态路由。该方案无法自然出现在 RuoYi 动态菜单中，且绕过现有权限体系。

## Risks / Trade-offs

- [Risk] 星形棋盘坐标渲染在窄屏上空间较紧 → 使用固定棋盘比例、稳定落点尺寸和容器滚动，优先保证点击准确性。
- [Risk] 连续跳跃状态容易造成用户不知道如何结束回合 → 页面在跳跃链状态展示“结束回合”按钮，并在侧边状态区明确提示。
- [Risk] 菜单 ID 与手工数据冲突 → 脚本复用扫雷已使用的 `休闲游戏` 目录 ID，并选择新的未占用子菜单 ID，同时使用 `on duplicate key update`。
- [Risk] 没有现成前端单元测试框架 → 沿用扫雷做法，使用 Node 原生断言测试纯逻辑模块，不为单个休闲游戏引入 Jest 或 Vitest。
- [Risk] 刷新页面会丢失对局 → 首版明确不做持久化，刷新后重新开始是可接受行为。

## Migration Plan

1. 部署前端构建产物。
2. 对已有数据库执行 `sql/chinese_checkers_menu.sql`，补充“休闲游戏 / 跳棋”菜单和 `common` 角色授权。
3. 对新环境可将相同菜单数据合入初始化流程，或在初始化后执行增量脚本。

回滚时删除新增前端文件，并删除对应 `sys_menu` 和 `sys_role_menu` 菜单数据即可。本变更不写入业务数据，不涉及业务数据迁移回滚。
