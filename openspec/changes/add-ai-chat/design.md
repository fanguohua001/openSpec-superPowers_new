## Context

当前项目是 RuoYi-Vue 3.9.1 风格的前后端分离管理系统。后端采用 Maven 多模块，业务 Controller 主要位于 `ruoyi-admin`，系统业务位于 `ruoyi-system`，通用工具位于 `ruoyi-common`。前端位于 `ruoyi-ui`，使用 Vue 2、Element UI 和 RuoYi 现有权限/路由体系。

现有系统已经具备用户、角色、菜单、权限、系统参数、Redis、MyBatis、代码生成等基础能力，但没有 AI 聊天相关业务。用户选择的首版范围是：兼容 OpenAI 格式的第三方服务、会话持久化、流式返回、模型配置存放在 `sys_config`、菜单入口放在“系统工具”下。

当前工作区已有与本次无关的配置修改：后端端口、数据库名/密码、前端代理端口。这些配置保持现状，不作为本变更的一部分覆盖。

## Goals / Non-Goals

**Goals:**

- 在“系统工具”下提供 AI 聊天页面。
- 支持当前用户自己的会话列表、消息历史和会话删除。
- 支持发送消息并以 `text/event-stream` 流式返回助手回复。
- 持久化用户消息、助手消息、生成状态和失败原因。
- 从 `sys_config` 读取模型服务配置。
- 默认支持 OpenAI Chat Completions 兼容流式协议。
- 保持 Controller、Service、Mapper、前端 API 和页面边界清晰。

**Non-Goals:**

- 不做多模型服务管理页面。
- 不做知识库、文件上传、联网搜索或 RAG。
- 不做 token 级精确上下文裁剪。
- 不引入跨请求任务取消注册表。
- 不改造 RuoYi 的通用系统参数页面。
- 不引入和当前需求无关的框架级重构。

## Decisions

### Decision 1: 以 `ruoyi-system` 承载 AI 聊天业务

AI 聊天的 domain、mapper、service 放在 `ruoyi-system`，Controller 放在 `ruoyi-admin`。这延续了现有系统功能的组织方式，避免新增模块带来的 Maven 依赖和包扫描复杂度。

备选方案是新增 `ruoyi-ai` 模块。它边界更独立，但首版功能规模还不值得增加模块管理成本。后续如果 AI 能力扩大到知识库、工具调用、多模型管理，再拆模块更合适。

### Decision 2: 使用两张表保存会话和消息

新增 `ai_chat_session` 和 `ai_chat_message`。会话表保存用户、标题、模型和状态；消息表保存角色、内容、状态和失败原因。消息表冗余 `user_id`，便于用户隔离、查询和清理。

备选方案是只建消息表，通过特殊字段聚合会话。它表少，但会话列表、删除、标题更新和权限过滤都会变得绕。两张表更接近聊天业务模型。

### Decision 3: 模型配置读取 `sys_config`

首版使用 `sys_config` 保存 `ai.chat.baseUrl`、`ai.chat.apiKey`、`ai.chat.model`、`ai.chat.temperature`、`ai.chat.timeoutSeconds`。这符合用户选择，也能复用现有参数管理能力。

风险是 `apiKey` 属于敏感信息，而通用系统参数页面默认不是密钥管理器。因此 AI 聊天页面不展示密钥，后端仅在调用模型时读取真实值；如后续要提供专用配置页，需要单独设计脱敏、写入和权限。

### Decision 4: 引入轻量模型适配边界

Service 不直接拼第三方请求，而是依赖模型适配器接口。默认实现 OpenAI Chat Completions 兼容流式协议，输入为上下文消息和参数，输出为增量回调。

备选方案是直接在 Service 中写 OpenAI 兼容调用。它更短，但用户已确认第三方协议“不确定”，直接耦合会增加后续替换成本。轻量适配边界能控制复杂度，又保留扩展空间。

### Decision 5: 使用 SSE 作为后端到前端的流式协议

后端发送 `text/event-stream`，事件包括 `session`、`message`、`delta`、`done`、`error`。前端据此创建会话、追加文本、展示完成或错误状态。

备选方案是 WebSocket。它适合双向长连接，但首版只是“用户发送一次，助手流式返回一次”，SSE 更简单，权限和 HTTP 调试也更贴近现有系统。

### Decision 6: 首版按消息条数裁剪上下文

发送模型请求时默认取最近 20 条消息，仅发送 `role` 和 `content`。不做 token 精算和自动摘要。

备选方案是引入 tokenizer 或摘要策略。它更精确，但依赖模型差异较大，且会扩大首版范围。消息条数限制足以保证首版可控。

## Risks / Trade-offs

- `apiKey` 存在 `sys_config` 中可能被高权限管理员查看 → 首版不在 AI 聊天页面展示密钥；后续如需专用配置页，单独设计脱敏和权限。
- 第三方模型服务不完全兼容 OpenAI SSE → 默认适配器只承诺 OpenAI 兼容协议；非兼容服务通过新增适配器处理。
- 浏览器或代理缓冲导致流式体验不明显 → 使用 `text/event-stream` 并避免后端一次性聚合；验证阶段必须确认前端能看到增量。
- 用户中断连接时后端无法可靠区分主动停止和网络异常 → 首版以保存失败状态或部分内容为准，不做复杂取消任务注册表。
- 消息过长可能超出模型上下文 → 首版按最近 20 条限制；后续再加 token 裁剪或摘要。
- AI 回复内容不可控 → 首版只提供通用聊天能力，不将回复直接写入业务数据；后续业务自动化需要独立审核流程。

## Migration Plan

1. 执行 SQL 初始化：
   - 创建 `ai_chat_session`。
   - 创建 `ai_chat_message`。
   - 写入 `sys_config` 默认配置项。
   - 写入 `sys_menu` 菜单和权限。
2. 部署后端代码。
3. 部署前端代码。
4. 管理员配置真实模型服务参数。
5. 使用管理员账号验证菜单、页面、发送、流式返回和历史记录。

回滚策略：

- 前端回滚后菜单可通过 SQL 或后台菜单管理禁用。
- 后端回滚前应停止使用 AI 聊天页面。
- 数据表为新增表，不影响现有业务表；如需要完全回滚，可备份后删除 AI 聊天相关表、菜单和配置。

## Open Questions

- 第三方模型服务最终是否完全兼容 OpenAI Chat Completions SSE。如果不兼容，需要新增对应适配器。
- 停止生成后的助手消息状态采用“失败并保存部分内容”还是“正常保存部分内容并标注已停止”。实现计划阶段需要选定一个策略并测试。
- `sys_config` 中 `apiKey` 的长期管理方式是否需要专用脱敏配置页。首版不做。
